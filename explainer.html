Section 01 — Architecture

Deux programmes,
une seule intelligence

NS-3 est le simulateur réseau (physique). Python est le cerveau RL (décisionnel). Ils communiquent via un socket TCP. Aucun ne peut fonctionner seul.

Section 02 — Flux de Messages

Chaque échange
décortiqué

NS-3 échange 5 types de messages JSON avec Python sur le socket TCP. Chaque type a un handler Python dédié dans FDQNServer.dispatch().

1
📤 NS-3 → Python
Requête PEPM batch
NS-3 envoie l'historique énergétique de tous les nœuds vivants en une seule requête (réduction du nombre de round-trips TCP).
{"type": "pepm_batch", "nodes": [{"id":42,"energy":0.94,"hist_len":10}, {"id":17,"energy":0.71,"hist_len":10}, ...]}
📥 Python → NS-3
Réponse PEPM batch
Python retourne un dictionnaire node_id → risque ∈ [0,1] pour tous les nœuds. Seuil d'alerte : PEPM_RISK_THRESHOLD = 0.7.
{"risks": {"42":0.23, "17":0.68, ...}, "status": "ok"} // risque par nœud [0,1] // > 0.7 = nœud évité dans routage // PEPM_RISK_THRESHOLD = 0.7
2
📤 NS-3 → Python
Demande d'action ADDQN
NS-3 envoie l'état normalisé du nœud i + la liste de ses voisins disponibles.
{"type": "action", "node_id": 42, "state": [0.94, 0.72, 0.23, 0.87, 0.00, 0.45, 0.88, 1.00, 0.31, 0.10], "neighbors": [17, 83, 105, 201]} // état = [E/E0, d_sink, σ_PEPM, voisins, isCH, TX, E_vois, frac_vivants, d_CH, recluster] (STATE_DIM=10)
📥 Python → NS-3
Action ADDQN
Python retourne l'ID du meilleur next hop selon la Q-table de l'agent.
{"action": 2, "next_hop": 105, "q_value": 0.847} // action=2 = 3ème voisin (index) // next_hop=105 = ID NS-3
3
📤 NS-3 → Python
Soumission de récompense
NS-3 informe Python du résultat de l'action. Python met à jour la Q-table.
{"type": "reward", "node_id": 42, "action": 2, "reward": 0.312, "next_state": [0.93, 0.72, 0.71, 0.86, 0.00, 0.46, 0.87, 1.00, 0.30, 0.12], // STATE_DIM=10 "done": false}
📥 Python → NS-3
Accusé d'apprentissage
Python confirme la mise à jour et retourne la perte d'apprentissage (loss).
{"ok": true, "loss": 0.0124, "aggregation_ok": 0, "federated": false, "fed_round": 42} // aggregation_ok=1 → NS-3 incrémente // m_fedRound (uniquement si FedAvg réel)
4
📤 NS-3 → Python
Envoi de topologie IFO
À chaque recluster IFO, NS-3 envoie la topologie clusters mise à jour pour que Python adapte la fédération.
{"type": "topology", "clusters": [{"ch":5,"members":[12,34,67]}, {"ch":18,"members":[2,91,105]}, ...]}
📥 Python → NS-3
Accusé de réception topologie
Python met à jour self.clusters pour adapter la prochaine agrégation FedAvg aux clusters actifs réels.
{"ok": true, "n_clusters": 23, "aggregation_ok": 0} // aggregation_ok=1 seulement après // une vraie agrégation FedAvg [FIX-F]
Section 03 — Rôles & Responsabilités

Zéro duplication :
chaque ligne a un rôle unique

NS-3 et Python font des choses fundamentalement différentes. Il n'y a aucune logique dupliquée. Voici la répartition exacte.

Responsabilité Qui le fait ? Pourquoi pas l'autre ? Preuve dans le code
Physique radio
propagation, RSSI, portée		 C++ NS-3 uniquement Python ne connaît pas les modèles physiques radio (Friis, Log-distance). NS-3 est construit pour ça. WifiPhy, FriisPropagationLoss
Consommation énergie
modèle batterie réaliste		 C++ NS-3 uniquement NS-3 modélise l'énergie à la milliseconde. Python ne voit que la valeur résiduelle, pas le modèle. BasicEnergySource, WifiRadioEnergyModel
Routage des paquets
transmission UDP réelle		 C++ NS-3 uniquement Python choisit vers qui envoyer. NS-3 envoie vraiment les paquets avec délais, pertes, collisions. UdpClientServer, OnOffApplication
Optimisation IFO
sélection des Cluster Heads		 C++ NS-3 uniquement IFO doit accéder aux états réels NS-3 (énergie live, distances). Python ne peut pas faire ça en temps réel. ifo.Run(nodeStates), calcFitness()
Réseau de neurones LSTM
prédiction risque PEPM		 PY Python uniquement Les libs deep learning (PyTorch/TF) n'existent pas en C++. NS-3 envoie juste l'énergie, Python fait le calcul. LSTM(input_size=1, hidden=64, window=10), TTD score
Double DQN (ADDQN)
apprentissage par renforcement		 PY Python uniquement Le DQN nécessite PyTorch, replay buffer, optimiseur Adam. Impossible en C++. NS-3 donne l'état, Python apprend. DQNAgent.select_action(), .learn()
Agrégation fédérée
FedAvg, méta-adaptation		 PY Python uniquement FedAvg = moyenne des poids PyTorch. Complètement impossible en C++. NS-3 ne sait même pas que FedAvg existe. fed_avg(weights_list), θ_global
Calcul de la récompense
R = f(PDR, E, délai)		 Les deux Calcul partagé NS-3 calcule les composantes réelles (PDR, ΔE mesurée). Python les combine selon la formule RL et met à jour Q. reward = 0.45·pdr + 0.20·E_norm − 0.10·delay − 0.10·risk + 0.15·hier
Sérialisation JSON
encodage/décodage messages		 Les deux Chaque côté son JSON NS-3 sérialise ses états en JSON (strings C++). Python désérialise et répond en JSON. C'est le seul point commun. json.dumps() / json.loads() / std::string
Section 04 — Apprentissage ADDQN

Comment l'agent
apprend à router ?

L'agent ADDQN apprend par essais-erreurs. À chaque action, il reçoit une récompense. La Q-table enregistre "quelle action est bonne dans quel état".

État S

Vecteur de 10 valeurs normalisées [0,1] : énergie, distance sink, risque PEPM, voisins vivants, rôle CH, transmissions, énergie voisins, fraction vivants, distance CH, recluster count

→

ADDQN

Double DQN : Q_online prédit, Q_target calcule la cible. Évite la surestimation.

→

Action A

Index du meilleur voisin (next hop). NS-3 route le paquet vers ce nœud.

Fonction de Récompense
R = λ₁·PDR + λ₂·E_norm − λ₃·Délai − λ₄·Risque + λ₅·Hier
λ₁ = 0.45
PDR
Livraison paquet : +1 si livré, −1 si non livré. Terme dominant (objectif cible 99%).
λ₂ = 0.20
+E_norm
Récompense l'énergie résiduelle normalisée. Plus un nœud a d'énergie, plus le routage via lui est valorisé.
λ₃ = 0.10
−Délai
Pénalise les chemins longs (distance next hop → sink). Favorise les routes directes.
λ₄ = 0.10
−Risque
Pénalise le score PEPM. Évite les routes via des nœuds proches de l'épuisement.
λ₅ = 0.15
+Hier
Bonus hiérarchie LEACH : récompense le respect de la structure cluster head / membre.

La mise à jour Double DQN :

# Cible TD (Temporal Difference)
y = r + γ · Q_target(s', argmax_a Q_online(s', a))

# Mise à jour des poids du réseau online
L = (y − Q_online(s, a))²
θ_online ← θ_online − α · ∇L

# Mise à jour hard du réseau target (toutes les 100 steps)
if step % 100 == 0: θ_target ← θ_online

# γ=0.99 (horizon futur) | α=3e-4 (Adam LR) | TARGET_UPDATE=100 steps (hard copy)

Section 06 — Chronologie d'une Simulation

Ce qui se passe
seconde par seconde

Voici la séquence exacte des événements pendant les 3500 secondes de simulation.

t = 0s — INITIALISATION
IFO Clustering initial
NS-3 place 300 nœuds aléatoirement. IFO calcule la fitness de chaque nœud (énergie + distance + densité) et sélectionne les 30 meilleurs comme Cluster Heads. Topologie exportée dans fdqnte_topology.csv.
t = 1s — CONNEXION
SOCKET NS-3 se connecte à Python
RLBridge ouvre une connexion TCP vers 127.0.0.1:5555. Python confirme. Les agents ADDQN sont instanciés pour les 300 nœuds. RLBridge confirme la connexion.
t = 2s — PREMIER STEP RL
ADDQN Première décision de routage
Pour chaque nœud vivant : NS-3 envoie l'état → Python calcule le risque PEPM → Python choisit le next hop via Double DQN (ε=0.9 = 90% aléatoire au début) → NS-3 applique la route.
t = 5s à 3500s — BOUCLE RL (toutes les 5s)
ADDQN 159 753 steps d'apprentissage
Toutes les 5s : pour chaque nœud vivant, NS-3 envoie l'état (STATE_DIM=10) → Python sélectionne l'action via Double DQN (ε décroît de 0,9 à 0,1) → récompense → entraînement par batch de 64. 159 753 steps au total jusqu'à t=3 500s.
t = 50s, 100s, 150s... — STATS
CSV Enregistrement des métriques
Toutes les 5s (RL_STEP_INTERVAL) : RlStep() calcule les métriques (énergie, PDR, délai, PEPM) et écrit une ligne dans fdqnte_energy.csv. 70 lignes pour 3 500s (une par round de 50s). Les fichiers comparison_metrics.csv et fdqnte_rl_history.json sont générés en fin de simulation.
t = 50s, 100s... (côté Python) — FEDAVG
FED 3 191 rondes d'agrégation fédérée confirmées
Toutes les FED_PERIOD=50 steps RL (≈ 250s simulées), _run_federation() est lancé dans un thread séparé. Il agrège les poids θ de tous les CH actifs par FedAvg et retourne aggregation_ok=1 → NS-3 incrémente m_fedRound. Total : 3 191 rondes confirmées sur 3 500s.
t = 100s, 200s... — RE-CLUSTERING
IFO Re-clustering adaptatif
Toutes les 100s, IFO recalcule la fitness en tenant compte de l'énergie résiduelle réelle. Les Cluster Heads peuvent changer si un nœud s'est affaibli. Garantit l'équilibre de charge.
t = 3 500s — FIN
RÉSULTAT Rapport final
Résultats finaux : 30 nœuds vivants / 270 morts, PDR_RL=95,7%, PDR_NS3=100%, E_moy=0,049J, E_cons=358,5J, délai=17,83ms. FND=2097s, HND=2627s, LND=3397s. Fichiers exportés : fdqnte_energy.csv, fdqnte_summary.csv, comparison_metrics.csv, fdqnte_rl_history.json.

Section 08 — Glossaire

Les termes
expliqués simplement

WSN — Wireless Sensor Network
Réseau de 300 capteurs IoT sans fil qui mesurent des données (température, humidité...) et les envoient à une station de base (Sink). Chaque capteur a une batterie limitée.
IFO — Isoptera Foraging Optimization
Algorithme inspiré du comportement des termites. Chaque nœud calcule sa "fitness" (énergie + distance + voisins). Les 30 meilleurs deviennent Cluster Heads.
ADDQN — Adaptive Double DQN
Variante du Deep Q-Network avec deux réseaux de neurones (online + target) pour éviter la surestimation des Q-values. "Adaptive" = epsilon décroît automatiquement.
PEPM — Predictive Energy Proactive Maintenance
LSTM qui prédit si un nœud risque d'épuiser sa batterie. Si σ > 0.7, l'ADDQN évite ce nœud dans ses routes. Prévention proactive des pannes.
FedMeta-DRL — Federated Meta Deep RL
Apprentissage fédéré : chaque agent apprend localement, puis FedAvg moyenne leurs poids. Cela accélère l'apprentissage global sans partager les données brutes.
Socket TCP
Canal de communication entre NS-3 (C++) et Python. NS-3 est le client, Python est le serveur. Les messages JSON sont échangés en temps réel pendant la simulation.
Epsilon (ε) — Exploration vs Exploitation
Au début (ε_max=0.9), l'agent choisit 90 % aléatoirement (exploration). Avec le temps (ε_min=0.1), il exploite sa Q-table. Décroissance linéaire : ε(t+1) = ε(t) − EPSILON_DELTA (δ=0.002), atteignant 0.1 après ≈ 400 steps. Pas de décroissance exponentielle.
PDR — Packet Delivery Ratio
Pourcentage de paquets qui arrivent au sink. PDR=100% signifie qu'aucun paquet n'a été perdu. C'est la métrique principale de qualité du routage.
FND — First Node Death
Round où le premier nœud épuise sa batterie. Plus FND est élevé, meilleur est le système. Avec E=1.2J et 3500s de simulation, FND survient à t=2097s (premier nœud mort).
Q-value
Valeur estimée de la récompense future si l'agent choisit l'action A dans l'état S. La Q-table mémorise ces valeurs. L'agent choisit toujours l'action avec la Q-value maximale.
